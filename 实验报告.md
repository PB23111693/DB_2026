## 一、系统设计

### 1.1 需求分析

本学籍管理系统旨在构建一个功能完整的 Web 应用，覆盖学生学籍管理的典型业务场景。前期设计中，要求系统至少需满足以下功能需求：

1. **学生基本信息管理**：增、删、改、查学生基本信息（学号、姓名、性别、出生日期、民族、政治面貌、籍贯、身份证号、联系电话、电子邮箱、入学年份、当前状态等信息）；支持照片、简历等文件上传与查看。
2. **专业变更管理**：记录学生专业变更历史（原专业、新专业、变更日期、审批文号、变更原因）。
3. **奖惩情况管理**：记录奖励信息（名称、级别、获奖日期、颁奖单位、证书文件）和处罚信息（类型、日期、原因、解除日期）；支持证明材料的上传。
4. **课程管理**：维护课程信息（编号、名称、学分、学时、开课院系、课程性质）；支持课程大纲文件上传。
5. **课程成绩管理**：录入、修改、查询学生各科成绩；成绩统计分析；成绩单PDF导出。
6. **用户与权限管理**：系统用户分为管理员、教师、学生三类。管理员拥有全部权限；教师可管理所授课程成绩、查看学生信息；学生只能查看本人信息与成绩。
7. **图片/视频/文件管理**：统一存储、类型过滤、大小限制、预览下载；文件与对应实体关联，保证数据一致性。
8. **其余实验要求**：必须包含存储过程、函数、事务、触发器四种数据库高级对象，实现特定业务逻辑。

在实际开发过程中，除了严格遵循前文所列的功能外，本人还在工程实现和应用场景上进行了多项功能扩展与细节优化，使得最终成果更加贴近真实教务管理需求。以下分别从这两方面进行补充说明。

#### 1.1.1 扩展功能

**1. 教师管理与教学工作台**
原始需求中仅笼统提及“教师可管理所授课程成绩、查看学生信息”，实际系统为教师角色构建了完整的工作界面：
- **管理员端教师管理**：管理员不仅能在“用户管理”中增删改教师账号，还拥有独立的“教师管理”视图，可查看每位教师的姓名、工号、所属院系，并实时显示该教师当前所授的所有课程（以标签列表形式展示），支持一键编辑或删除教师。
- **教师工作台**：教师登录后，导航栏自动切换为教师专属菜单，包含“我的课程”入口。教师可浏览自己教授的课程列表、每门课程的选课学生人数，点击课程可查看详细的学生名单，并在该页面对每个学生逐一录入或修改成绩，无需跳转到通用的成绩管理页面。这极大方便了教师的日常工作流。

**2. 学生自主选课管理**
原需求并未涉及学生自主选课，仅定义了管理员通过系统为学生分配课程的功能。实际系统为学生角色增加了“选课中心”和“我的课程”两个模块：
- 学生登录后可浏览所有已分配教师的课程信息（尚未分配教师的课程不在列表中展示），自主选择心仪课程并确认选课；也可对已选但尚未有成绩的课程进行退选。
- 选课与退选操作会实时与数据库交互，退选时若课程已有成绩则拒绝退选，保证数据一致性。
- “我的课程”页面则清晰展示学生当前学期已选课程、授课教师、学分及成绩，形成完整的学习状况总览。

**3. 基础数据（院系/专业/班级）前端维护**
原始E-R图中包含了院系、专业、班级实体，但未明确要求前端管理界面。实际系统为管理员提供了完整的“基础数据”菜单，可对院系、专业、班级进行图形化的增删改查操作，且班级辅导员、专业负责人等字段均与教师工号关联，从下拉框中选择已有教师，保证了数据引用的准确性。

**4. 学生账户同步创建**
添加学生信息时，管理员可勾选“同步创建登录账号”，系统会自动生成与学号同名的用户（初始密码为123456），并分配学生角色，一次操作完成学籍录入和账号开通，简化了管理员工作。

#### 1.1.2 优化改进

在基本功能实现后，开发过程中针对发现的问题进行了大量细节优化，主要包括：

- **成绩录入校验与格式化**：
  - 成绩输入框添加 `min="0"` 限制，后端同步进行非负判断，彻底禁止录入负数成绩。
  - 模板中所有成绩、学分等数字均使用 `"%.1f"|format` 或 `"%.2f"|format` 格式化，确保显示简洁（如 4.0 而非 4.00000），GPA 保留两位小数。
  - 成绩录入时仅更新用户实际填写的字段，若只填写了部分成绩（如平时和期末），则不会将未填写的原有成绩清空，避免了误操作导致的数据丢失。

- **专业变更防呆**：存储过程 `sp_change_major` 内增加了新旧专业相同的判断，若学生申请的“新专业”与当前专业一致，则拒绝操作并给出明确提示。

- **数据库字符集统一**：所有建表语句和存储过程参数显式指定 `utf8mb4_unicode_ci`，解决了因字符集冲突导致的专业变更等操作报错问题。

- **辅导员字段关联教师工号**：将班级表中 `counselor` 字段改为 `counselor_id`，并与教师工号表联动，避免纯文本输入造成的数据不一致。

- **文件生命周期管理**：删除学生或课程时，关联的照片、简历、大纲等服务器文件（含缩略图）会被自动清理，防止无用文件累积。

- **教师与学生自助密码修改**：增加独立的密码修改页面，任何角色均可自行修改密码，无需管理员介入。

- **课程列表过滤**：未分配教师的课程不会出现在学生选课列表中，避免学生误选无人授课的课程。

- **权限与安全精细化控制**：
  - 最后一个管理员不能被降级为教师或学生，也不能被删除，防止系统失去管理员。
  - 管理员修改自身角色后，会强制退出登录，避免出现权限不一致的会话。
  - 教师只能为自己负责课程的学生录入成绩，且学生必须先选修该课程，杜绝了教师“为未选课的学生赋分”的越权行为。
  - 所有文件上传均通过扩展名白名单和大小限制进行过滤，防止恶意文件上传。


这些扩展与优化使得最终的学籍管理系统不仅在功能上覆盖了需求，更在易用性、安全性和数据一致性上达到了较高的实用标准。

#### 1.2.2 新增实体说明

#### 1. STUDENT_SUMMARY（学生学业汇总）

**作用**：缓存学生的实时 GPA、已获学分以及是否存在未解除处分，避免每次查询都进行复杂的聚合计算。该表由触发器自动维护，应用层只读不写，保证了汇总数据与实际成绩、处分状态的强一致性。

| 属性名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| `stu_id` | VARCHAR(20) | **主键**，外键引用`student.stu_id` | 关联的学生学号 |
| `total_credits` | DECIMAL(5,1) | 默认值 0 | 已获得的学分总数（要求总评成绩≥60，否则视为不及格） |
| `gpa` | DECIMAL(4,2) | 默认值 0.00 | 平均学分绩点 |
| `has_punish` | BOOLEAN | 默认值 FALSE | 是否尚有未解除的处分记录 |

**与原有实体的联系**：
- 与 `STUDENT` 实体是 **1 : 1** 关系（一个学生对应一条汇总记录，`student_summary.stu_id` 同时是外键和主键）。
- 汇总数据通过触发器自动生成，不单独进行增删改操作。

#### 2. USERS（用户认证与角色）

**作用**：支持系统登录与三级权限（管理员、教师、学生）。通过 `related_id` 字段与学生学籍或教师工号相关联，实现用户身份与实际业务数据（学生、课程）的绑定。密码使用安全哈希存储。

**表结构（属性列表）**：

| 属性名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| `user_id` | INT | **主键**，自增 | 内部唯一标识 |
| `username` | VARCHAR(30) | 唯一，非空 | 登录用户名 |
| `password_hash` | VARCHAR(255) | 非空 | 密码哈希值（由 werkzeug 生成） |
| `role` | VARCHAR(10) | 非空 | 角色：admin / teacher / student |
| `name` | VARCHAR(30) | 可空 | 真实姓名（便于显示） |
| `dept_id` | VARCHAR(10) | 可空，外键引用`department.dept_id` | 教师所属院系 |
| `related_id` | VARCHAR(20) | 可空 | 关联学号（学生）或工号（教师） |
| `last_login` | DATETIME | 可空 | 最近一次登录时间 |

**与原有实体的联系**：
- 与 `STUDENT` 实体通过 `related_id` 间接关联：`users.related_id = student.stu_id`（学生角色）
- 与教师工号（未作为独立表，但通过 `Course.teacher_id` 关联）关联：`users.related_id = course.teacher_id`（教师角色）
- 与 `DEPARTMENT` 通过 `users.dept_id` 关联，表示教师所属院系。
- 该表为典型的 **1 : 1** 或 **0..1** 关系：一个学生/教师最多拥有一个登录账号。

新增的两个表均紧密依附于原有ER图的核心实体，未破坏原有的关系基数。它们分别解决了性能优化和应用安全需求，是为了使数据库系统贴近实用需求而做的必要补充。在数据库层面，它们同样满足3NF，不存在冗余或传递依赖。

## 二、数据库核心设计与实现

### 2.1 数据库总体设计

数据库采用 MySQL 8.0，字符集统一为 `utf8mb4_unicode_ci`，共创建 12 张表：

- **基础数据**：`department`、`major`、`classes`
- **学生相关**：`student`、`major_change`、`award`、`punishment`
- **课程与成绩**：`course`、`enrollment`、`score`
- **汇总与用户**：`student_summary`、`users`

所有表均设有主键，外键约束保证参照完整性。此外，定义了 2 个函数、4 个触发器、2 个存储过程，将核心业务逻辑植入数据库层。

### 2.2 核心表结构详解

**1. student（学生表）**
```sql
CREATE TABLE student (
    stu_id VARCHAR(20) PRIMARY KEY,
    name VARCHAR(30) NOT NULL,
    gender CHAR(1),
    birth DATE,
    ethnicity VARCHAR(20),
    political_status VARCHAR(20),
    native_place VARCHAR(50),
    id_card CHAR(18),
    phone VARCHAR(20),
    email VARCHAR(50),
    address VARCHAR(100),
    enrollment_year YEAR,
```
```sql
    status VARCHAR(10) DEFAULT '在读',
    photo_path VARCHAR(200),
    resume_path VARCHAR(200),
    class_id VARCHAR(10) NOT NULL,
    FOREIGN KEY (class_id) REFERENCES classes(class_id)
```

学号是主键，直接标识一名学生。`class_id` 外键关联到班级，从而可以推导出专业和院系，消除了冗余。`photo_path` 和 `resume_path` 存储服务器上的相对路径，便于文件管理。

**2. enrollment（选课表）**
```sql
CREATE TABLE enrollment (
    enroll_id INT AUTO_INCREMENT PRIMARY KEY,
    stu_id VARCHAR(20) NOT NULL,
    course_id VARCHAR(10) NOT NULL,
    semester VARCHAR(20) NOT NULL,
    enroll_date DATE,
    FOREIGN KEY (stu_id) REFERENCES student(stu_id),
    FOREIGN KEY (course_id) REFERENCES course(course_id),
    UNIQUE KEY uk_sc (stu_id, course_id, semester)
);
```
该表解决了学生与课程多对多关系，增加 `semester` 字段以支持不同学期。唯一键防止重复选课。选课记录是成绩存在的先决条件。

**3. score（成绩表）**
```sql
CREATE TABLE score (
    score_id INT AUTO_INCREMENT PRIMARY KEY,
    enroll_id INT NOT NULL UNIQUE,
    regular_score DECIMAL(5,2),
    final_score DECIMAL(5,2),
    total_score DECIMAL(5,2),
    exam_method VARCHAR(10),
    makeup_score DECIMAL(5,2),
    FOREIGN KEY (enroll_id) REFERENCES enrollment(enroll_id)
);
```
成绩与选课一对一关系。这种设计避免了在成绩表中再次存储学号和课程号，保持数据一致性且符合 3NF。补考成绩单独保留。

**4. student_summary（学生学业汇总表）**
```sql
CREATE TABLE student_summary (
    stu_id VARCHAR(20) PRIMARY KEY,
    total_credits DECIMAL(5,1) DEFAULT 0,
    gpa DECIMAL(4,2) DEFAULT 0,
    has_punish BOOLEAN DEFAULT FALSE,
    FOREIGN KEY (stu_id) REFERENCES student(stu_id)
);
```
该表由触发器自动维护，不允许应用层直接写入。`total_credits` 仅累加成绩 >60 的课程学分，`gpa` 调用函数实时计算。

**5. users（用户认证与角色表）**
```sql
CREATE TABLE users (
    user_id INT AUTO_INCREMENT PRIMARY KEY,
    username VARCHAR(30) NOT NULL UNIQUE,
    password_hash VARCHAR(255) NOT NULL,
    role VARCHAR(10) NOT NULL,   -- admin/teacher/student
    name VARCHAR(30),
    dept_id VARCHAR(10),
    related_id VARCHAR(20),
    last_login DATETIME,
    FOREIGN KEY (dept_id) REFERENCES department(dept_id)
);
```
通过 `related_id` 与教师工号或学生学号关联，`role` 字段标识权限。密码采用 `werkzeug.security.generate_password_hash` 加密。

### 2.3 高级数据库对象（存储过程、函数、触发器、事务）

为了将核心业务规则固化到数据库层，减少应用层的重复逻辑并保证数据一致性，本系统在 MySQL 中设计了两个用户自定义函数、五个触发器以及两个存储过程。这些对象相互配合，共同构成了系统数据完整性的关键防线。

#### 2.3.1 函数

**fn_calc_student_gpa —— 计算学生平均学分绩点**

```sql
CREATE FUNCTION fn_calc_student_gpa(p_stu_id VARCHAR(20))
RETURNS DECIMAL(5,2)
DETERMINISTIC
READS SQL DATA
BEGIN
    DECLARE total_credits DECIMAL(5,1);
    DECLARE weighted_score DECIMAL(10,2);
    SELECT SUM(c.credit),
           SUM(c.credit * CASE
               WHEN s.total_score >= 90 THEN 4.0
               WHEN s.total_score >= 85 THEN 3.7
               WHEN s.total_score >= 82 THEN 3.3
               WHEN s.total_score >= 78 THEN 3.0
               WHEN s.total_score >= 75 THEN 2.7
               WHEN s.total_score >= 72 THEN 2.3
               WHEN s.total_score >= 68 THEN 2.0
               WHEN s.total_score >= 65 THEN 1.7
               WHEN s.total_score >= 64 THEN 1.5
               WHEN s.total_score >= 61 THEN 1.3
               WHEN s.total_score >= 60 THEN 1.0
               ELSE 0 END)
    INTO total_credits, weighted_score
    FROM score s
    JOIN enrollment e ON s.enroll_id = e.enroll_id
    JOIN course c ON e.course_id = c.course_id
    WHERE e.stu_id = p_stu_id AND s.total_score IS NOT NULL;
    IF total_credits IS NULL OR total_credits = 0 THEN RETURN 0;
    ELSE RETURN ROUND(weighted_score / total_credits, 2);
    END IF;
END
```

该函数是学生学业评估的核心，它根据学生所有已录入成绩，严格按照设定的百分制与绩点对应表，计算加权平均绩点（GPA）。函数首先通过子查询关联 `score`、`enrollment` 和 `course` 表，获取该生所有有效成绩及对应课程的学分。然后利用 `CASE` 表达式将总评成绩映射为 4.0 制绩点。将每门课程的绩点乘以该课程的学分，累加得到加权分数之和，再除以已修课程的总学分，最终结果保留两位小数。函数被声明为 `DETERMINISTIC` 并带有 `READS SQL DATA` 特性，表示对于相同的输入参数总是返回相同结果，且仅读取数据，这有助于 MySQL 查询优化器在某些上下文中进行优化。

`fn_calc_student_gpa` 在多个场景中被调用：①成绩插入或更新的触发器会自动调用它来更新 `student_summary` 表中的 GPA；②成绩删除后，应用层通过手动执行 SQL 语句调用该函数重新计算 GPA；③当需要查询某个学生的学业状况时，可直接在 SQL 查询中调用。通过将绩点计算逻辑封装为函数，避免了在应用层和前端分别实现相同规则，保证了 GPA 计算口径的唯一性和一致性。

**fn_can_graduate —— 判断是否达到毕业要求**

```sql
CREATE FUNCTION fn_can_graduate(p_stu_id VARCHAR(20))
RETURNS BOOLEAN
DETERMINISTIC
READS SQL DATA
BEGIN
    DECLARE v_total DECIMAL(5,1);
    DECLARE v_gpa DECIMAL(5,2);
    SELECT total_credits, gpa INTO v_total, v_gpa FROM student_summary WHERE stu_id = p_stu_id;
    RETURN (v_total >= 130 AND v_gpa >= 2.0);
END
```

该函数返回一个布尔值，用于快速判断学生是否满足毕业要求。它直接从 `student_summary` 表读取该生的总学分和 GPA，检查总学分是否达到 130 分且 GPA 不低于 2.0。由于 `student_summary` 由触发器自动维护，因此该函数总是基于最新的成绩数据进行判断。毕业审核功能可直接在查询中使用此函数进行筛选，无需编写复杂条件，提升了代码可读性和可维护性。

#### 2.3.2 触发器

**trg_score_after_change / trg_score_after_update（成绩变更触发器）**

```sql
CREATE TRIGGER trg_score_after_change
AFTER INSERT ON score FOR EACH ROW
BEGIN
    DECLARE v_stu_id VARCHAR(20);
    SELECT stu_id INTO v_stu_id FROM enrollment WHERE enroll_id = NEW.enroll_id;
    UPDATE student_summary SET
        gpa = fn_calc_student_gpa(v_stu_id),
        total_credits = (SELECT COALESCE(SUM(c.credit),0)
                         FROM score s
                         JOIN enrollment e ON s.enroll_id = e.enroll_id
                         JOIN course c ON e.course_id = c.course_id
                         WHERE e.stu_id = v_stu_id AND s.total_score >= 60)
    WHERE stu_id = v_stu_id;
END
```
```sql
CREATE TRIGGER trg_score_after_update
AFTER UPDATE ON score
FOR EACH ROW
BEGIN
    DECLARE v_stu_id VARCHAR(20);
    SELECT stu_id INTO v_stu_id FROM enrollment WHERE enroll_id = NEW.enroll_id;
    UPDATE student_summary SET
        gpa = fn_calc_student_gpa(v_stu_id),
        total_credits = (SELECT COALESCE(SUM(c.credit),0)
                         FROM score s JOIN enrollment e ON s.enroll_id = e.enroll_id
                         JOIN course c ON e.course_id = c.course_id
                         WHERE e.stu_id = v_stu_id AND s.total_score >= 60)
    WHERE stu_id = v_stu_id;
END
```

这两个触发器分别在 `score` 表发生 INSERT 和 UPDATE 后自动执行。它们首先通过 `enrollment` 表获取该成绩所属的学生学号，然后调用 `fn_calc_student_gpa` 函数计算最新的 GPA，并通过一个子查询统计该生所有总评成绩 ≥60 分的课程学分之和，最后将这两个值更新到 `student_summary` 表中。这种设计使得学生每次成绩变动后，其学业汇总都会立即保持同步，完全无需应用层介入。无论在哪个页面录入或修改成绩，都能保证 `student_summary` 始终反映最新状态。

**trg_punishment_after_insert / trg_punishment_after_update（处分状态触发器）**

```sql
CREATE TRIGGER trg_punishment_after_insert
AFTER INSERT ON punishment
FOR EACH ROW
BEGIN
    IF NEW.lift_date IS NULL OR NEW.lift_date > CURDATE() THEN
        UPDATE student_summary SET has_punish = TRUE WHERE stu_id = NEW.stu_id;
    END IF;
END;
```
```sql
CREATE TRIGGER trg_punishment_after_update
AFTER UPDATE ON punishment
FOR EACH ROW
BEGIN
    IF NEW.lift_date IS NOT NULL AND NEW.lift_date <= CURDATE() THEN
        IF NOT EXISTS (SELECT 1 FROM punishment WHERE stu_id = NEW.stu_id AND (lift_date IS NULL OR lift_date > CURDATE())) THEN
            UPDATE student_summary SET has_punish = FALSE WHERE stu_id = NEW.stu_id;
        END IF;
    END IF;
END
```

这套触发器用于维护学生的处分标记。当插入一条新处分且其解除日期尚未到来时，将 `student_summary.has_punish` 置为 TRUE。当处分记录被更新（通常是因为管理员设置了解除日期）且该解除日期已到，触发器会进一步检查该生是否还有其他尚未解除的处分。只有在无其他未解除处分的情况下，才会将标记重置为 FALSE。

**trg_student_after_insert（学生记录初始化触发器）**

```sql
CREATE TRIGGER trg_student_after_insert
AFTER INSERT ON student
FOR EACH ROW
BEGIN
    INSERT INTO student_summary (stu_id) VALUES (NEW.stu_id)
    ON DUPLICATE KEY UPDATE stu_id = NEW.stu_id;
END
```

每当新增一名学生，该触发器会自动在 `student_summary` 表中插入一条初始记录，使其总学分和 GPA 从 0 开始。这样后续的成绩录入才能正常触发更新。使用 `ON DUPLICATE KEY UPDATE` 避免了因意外重复操作而导致触发器报错的问题。

#### 2.3.3 存储过程

**sp_change_major —— 专业变更事务**

```sql
CREATE PROCEDURE sp_change_major(
    IN p_stu_id VARCHAR(20), IN p_new_major_id VARCHAR(10),
    IN p_change_date DATE, IN p_approval_doc VARCHAR(50),
    IN p_reason VARCHAR(200), IN p_operator VARCHAR(20))
BEGIN
    DECLARE v_old_major_id VARCHAR(10);
    DECLARE v_new_class_id VARCHAR(10);
    DECLARE EXIT HANDLER FOR SQLEXCEPTION BEGIN ROLLBACK; RESIGNAL; END;
    START TRANSACTION;
    SELECT m.major_id INTO v_old_major_id
    FROM student st JOIN classes c ON st.class_id = c.class_id
    JOIN major m ON c.major_id = m.major_id
    WHERE st.stu_id = p_stu_id;
    IF v_old_major_id = p_new_major_id THEN
        SIGNAL SQLSTATE '45000' SET MESSAGE_TEXT = '新专业与当前专业相同，无法变更';
    END IF;
    SELECT class_id INTO v_new_class_id FROM classes WHERE major_id = p_new_major_id LIMIT 1;
    IF v_new_class_id IS NULL THEN
        SIGNAL SQLSTATE '45000' SET MESSAGE_TEXT = '新专业无可用班级';
    END IF;
    INSERT INTO major_change (stu_id, origin_major, new_major, change_date, approval_doc, reason, operator)
    VALUES (p_stu_id, v_old_major_id, p_new_major_id, p_change_date, p_approval_doc, p_reason, p_operator);
    UPDATE student SET class_id = v_new_class_id WHERE stu_id = p_stu_id;
    COMMIT;
END
```

专业变更是典型的业务操作，涉及多条 SQL 语句，必须保证原子性。存储过程将整个操作封装在一个显式事务内：首先获取学生当前专业，如果新旧专业相同则直接拒绝；接着查找新专业的默认班级，若不存在也拒绝；然后向 `major_change` 表插入变更历史记录，再更新 `student` 表的班级字段。存储过程声明了 `EXIT HANDLER FOR SQLEXCEPTION`，任何一步失败都会触发回滚并重新抛出异常，应用层可以捕获该异常并给用户明确的错误提示。此设计不仅保证了数据的一致性，还内置了防止重复变更的校验逻辑。

**sp_import_scores —— 批量导入成绩**

```sql
CREATE PROCEDURE sp_import_scores(
    IN score_data JSON,
    IN p_semester VARCHAR(20)
)
BEGIN
    DECLARE i INT DEFAULT 0;
    DECLARE n INT DEFAULT JSON_LENGTH(score_data);
    DECLARE v_stu_id VARCHAR(20);
    DECLARE v_course_id VARCHAR(10);
    DECLARE v_exam VARCHAR(10);
    DECLARE v_regular, v_final, v_total DECIMAL(5,2);
    DECLARE v_enroll_id INT;
    WHILE i < n DO
        SET v_stu_id = JSON_UNQUOTE(JSON_EXTRACT(score_data, CONCAT('$[', i, '].stu_id')));
        SET v_course_id = JSON_UNQUOTE(JSON_EXTRACT(score_data, CONCAT('$[', i, '].course_id')));
        SET v_regular = JSON_EXTRACT(score_data, CONCAT('$[', i, '].regular'));
        SET v_final = JSON_EXTRACT(score_data, CONCAT('$[', i, '].final'));
        SET v_total = JSON_EXTRACT(score_data, CONCAT('$[', i, '].total'));
        SET v_exam = JSON_UNQUOTE(JSON_EXTRACT(score_data, CONCAT('$[', i, '].exam')));
        SELECT enroll_id INTO v_enroll_id
        FROM enrollment
        WHERE stu_id = v_stu_id AND course_id = v_course_id AND semester = p_semester;
        IF v_enroll_id IS NULL THEN
            INSERT INTO enrollment (stu_id, course_id, semester, enroll_date)
            VALUES (v_stu_id, v_course_id, p_semester, CURDATE());
            SET v_enroll_id = LAST_INSERT_ID();
        END IF;
        INSERT INTO score (enroll_id, regular_score, final_score, total_score, exam_method)
        VALUES (v_enroll_id, v_regular, v_final, v_total, v_exam)
        ON DUPLICATE KEY UPDATE
            regular_score = v_regular,
            final_score   = v_final,
            total_score   = v_total,
            exam_method   = v_exam;
        SET i = i + 1;
    END WHILE;
END
```

该存储过程用于批量导入或迁移成绩数据。过程接收一个 JSON 数组和一个学期字符串。遍历数组时，对于每条成绩记录，它会先到 enrollment 表中查询指定学生、课程和学期的选课记录；若不存在，则自动创建一条选课记录（选课日期设为当天），再向 score 表插入或更新成绩（通过 ON DUPLICATE KEY UPDATE 实现存在即更新）。整个操作在一个事务中完成（由外部调用者控制），并会触发相应的触发器自动更新 student_summary 中的 GPA 和学分。此过程主要用于系统初始化、数据迁移或管理员批量导入成绩的场景，大幅简化了操作流程。

适用场景：系统初始化后，批量导入历史学期的成绩；从旧教务系统迁移数据时，将已有的选课及成绩一并导入；管理员在特殊情况下（如大面积补录）通过 SQL 脚本或 Python 脚本快速写入成绩。

本次实验中根目录中的 `import_scores.py` 脚本即为此存储过程的 Python 封装，用于测试和演示。

> 注：与日常成绩录入的区别：
日常业务中，前端成绩管理接口（score.manage）严格要求学生必须先选课，否则拒绝录入，而 sp_import_scores 作为后台维护工具，在导入时自动补全缺失的选课记录，是为了降低批量操作的复杂度，不供教师和学生日常使用，也不存在任何前端操作方式，因此两者不存在逻辑冲突。

#### 2.3.4 事务应用与小结

上述数据库对象协同工作，将核心业务逻辑深植于数据库层。触发器实现了数据变化的自动响应，函数封装了计算规则，存储过程则通过事务保证了复杂操作的原子性。这些高级特性的合理运用，不仅满足了课程设计对存储过程、函数、触发器和事务的四项硬性要求，更使得整个系统的数据严谨性和操作安全性得到了质的提升。


学生 GPA 排名则通过计算每个学生的加权总分与总学分的商获得，并按降序排列，前端用表格展示。

**成绩单 PDF 导出**在 `score.download_transcript` 路由中实现，调用 `services/pdf_service.py` 中的 `generate_transcript` 函数。该函数使用 `reportlab` 库动态绘制表格，并注册中文字体 `SimSun.ttf`，生成的 PDF 包含学生基本信息、所有课程成绩及绩点，末尾汇总显示 GPA。这满足了学籍管理中成绩单打印的实际需求。

### 4.3 学生自主选课与退选

为了增强学生的自主性，系统在 `routes/student.py` 中增加了 `enroll_course` 和 `toggle_enroll` 两个视图，以及相应的前端模板。

**选课中心 (`enroll_course`)**
展示所有已分配教师的课程（未分配教师的课程不显示），学生可以浏览课程信息，并通过表单提交选课请求。

```python
@bp.route('/enroll')
@login_required
def enroll_course():
    if current_user.role != 'student':
        abort(403)
    stu_id = current_user.related_id
    # 只显示有教师的课程
    all_courses = Course.query.filter(Course.teacher_id.isnot(None)).all()
    enrolled_ids = {e.course_id for e in Enrollment.query.filter_by(stu_id=stu_id).all()}
    return render_template('student/enroll.html',
                           courses=all_courses,
                           enrolled_course_ids=enrolled_ids)
```

**选课/退选操作 (`toggle_enroll`)**
通过同一路由接受 POST 请求，根据是否存在选课记录来决定执行选课还是退选。退选时，会检查该课程是否已录入成绩，若有成绩则阻止退选，保证数据完整性。

```python
@bp.route('/enroll/<course_id>', methods=['POST'])
def toggle_enroll(course_id):
    enrollment = Enrollment.query.filter_by(
        stu_id=current_user.related_id, course_id=course_id, semester='2024-2025-1'
    ).first()
    if enrollment:
        # 退选：有成绩则禁止
        if Score.query.filter_by(enroll_id=enrollment.enroll_id).first():
            flash('该课程已有成绩，无法退选', 'danger')
        else:
            db.session.delete(enrollment)
            db.session.commit()
            flash('退选成功', 'info')
    else:
        # 选课
        new_enroll = Enrollment(stu_id=..., course_id=..., semester=..., enroll_date=date.today())
        db.session.add(new_enroll)
        db.session.commit()
        flash('选课成功', 'success')
    return redirect(url_for('student.enroll_course'))
```

这种实现既赋予了学生选课的灵活性，又通过成绩存在性检查避免了数据不一致问题。

### 4.4 教师工作台

教师工作台由 `routes/teacher.py` 蓝图实现，提供了两个核心视图：

- **`dashboard`**：查询当前教师（通过 `current_user.related_id`）所负责的所有课程，并统计每门课程的选课学生人数。
- **`course_students`**：显示某门课程的学生名单及成绩，并提供行内成绩录入功能。

成绩录入部分同样遵循部分字段更新、教师课程归属校验、负数禁止等原则，与管理员端成绩管理逻辑保持一致，但交互上更专注于教师的日常教学场景。

### 4.5 专业变更管理（事务控制）

专业变更由 `sp_change_major` 存储过程完成，前端通过 `routes/major_change.py` 调用：

```python
with db.engine.begin() as conn:
    conn.execute(text("CALL sp_change_major(:stu, :new, :date, :doc, :reason, :op)"), params)
```

存储过程内部使用显式事务，并在检测到新旧专业相同或无可用班级时通过 `SIGNAL` 返回错误消息，应用层捕获后反馈给用户。这种设计将复杂的业务规则和事务控制封装在数据库层，降低了应用层的复杂度。

### 4.6 文件与安全管理

文件上传统一由 `upload` 蓝图处理，保存时通过 `services/file_service.py` 中的 `save_file` 函数生成 UUID 文件名，并存放于对应子目录。删除学生或课程时，会主动调用 `delete_file` 清理物理文件。文件类型和大小限制在 `config.py` 中配置，上传时校验，有效防范了恶意文件上传。
